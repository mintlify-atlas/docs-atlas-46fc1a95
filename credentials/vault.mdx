HashiCorp Vault is the most feature-rich credentials backend available for Concourse. The Concourse `web` node connects directly to a Vault server, authenticates using one of several supported methods, and performs KV lookups on behalf of running builds. Vault's lease system is respected: Concourse tracks expiry times returned by the API and refreshes cached secrets before they expire, so builds always receive a valid value without needing to re-authenticate.

## How authentication works

Concourse supports three authentication strategies against Vault, selected via the `CONCOURSE_VAULT_AUTH_BACKEND` variable on the `web` node:

<Accordion title="Token auth (simplest, not recommended for production)">
  Pass a long-lived Vault token directly. The token must have a policy granting read access to the `/concourse` path prefix. Token renewal is handled automatically, but static tokens can be revoked or expire without warning.

  ```bash
  CONCOURSE_VAULT_URL=https://vault.example.com
  CONCOURSE_VAULT_CLIENT_TOKEN=s.abcdefghijk1234
  ```
</Accordion>

<Accordion title="AppRole auth (recommended for production)">
  AppRole allows Concourse to authenticate with a `role_id` and `secret_id` pair, producing short-lived tokens that are renewed automatically. The `secret_id` can be wrapped (response-wrapped) for extra security.

  ```bash
  CONCOURSE_VAULT_URL=https://vault.example.com
  CONCOURSE_VAULT_AUTH_BACKEND=approle
  CONCOURSE_VAULT_AUTH_PARAM=role_id:your-role-id
  CONCOURSE_VAULT_AUTH_PARAM=secret_id:your-secret-id
  ```
</Accordion>

<Accordion title="TLS certificate auth (mutual TLS)">
  Concourse presents a client certificate and key during the TLS handshake. Vault maps the certificate's CN or SAN to a policy. This is the most secure option and requires no shared secret.

  ```bash
  CONCOURSE_VAULT_URL=https://vault.example.com
  CONCOURSE_VAULT_AUTH_BACKEND=cert
  CONCOURSE_VAULT_CLIENT_CERT=/etc/concourse/vault-client.crt
  CONCOURSE_VAULT_CLIENT_KEY=/etc/concourse/vault-client.key
  ```
</Accordion>

## Configuration reference

All Vault settings are passed as environment variables to the `concourse web` process, or as flags with the equivalent `--vault-` prefix.

<Accordion title="Connection settings">
  <ParamField body="CONCOURSE_VAULT_URL" type="string" required>
    Full URL of the Vault server, including scheme. Example: `https://vault.example.com:8200`
  </ParamField>

  <ParamField body="CONCOURSE_VAULT_PATH_PREFIX" type="string" default="/concourse">
    Root path under which all credential lookups are performed. Concourse appends the team, pipeline, and secret name to this prefix.
  </ParamField>

  <ParamField body="CONCOURSE_VAULT_SHARED_PATH" type="string">
    Optional path for secrets accessible to all teams and pipelines. Checked after team-scoped paths. Example: `shared`
  </ParamField>

  <ParamField body="CONCOURSE_VAULT_NAMESPACE" type="string">
    Vault Enterprise namespace to use for both authentication and secret lookups.
  </ParamField>

  <ParamField body="CONCOURSE_VAULT_LOOKUP_TEMPLATES" type="string">
    Comma-separated list of path templates used for credential lookup (relative to the path prefix). Defaults to `/{{.Team}}/{{.Pipeline}}/{{.Secret}}` and `/{{.Team}}/{{.Secret}}`. Override both templates together when customising the lookup hierarchy.
  </ParamField>

  <ParamField body="CONCOURSE_VAULT_LOGIN_TIMEOUT" type="duration" default="60s">
    How long Concourse will wait for the initial Vault login before returning an error.
  </ParamField>

  <ParamField body="CONCOURSE_VAULT_QUERY_TIMEOUT" type="duration" default="60s">
    Per-request timeout for secret reads.
  </ParamField>

  <ParamField body="CONCOURSE_VAULT_DISABLE_SRV_LOOKUP" type="boolean" default="false">
    When the Vault URL contains no port, the client performs a DNS SRV lookup by default. Set this flag to disable that behaviour. Has no effect when the URL includes an explicit port.
  </ParamField>

  <ParamField body="CONCOURSE_VAULT_ENABLE_KV_MOUNT_CACHE" type="boolean" default="false">
    Cache the KV mount version detection result to reduce the number of preflight requests sent to Vault on every secret lookup.
  </ParamField>
</Accordion>

<Accordion title="TLS settings">
  <ParamField body="CONCOURSE_VAULT_CA_CERT" type="string">
    Path to a PEM-encoded CA certificate file used to verify the Vault server's TLS certificate. Required when Vault uses a private or self-signed CA.
  </ParamField>

  <ParamField body="CONCOURSE_VAULT_CA_PATH" type="string">
    Path to a directory of PEM-encoded CA certificate files.
  </ParamField>

  <ParamField body="CONCOURSE_VAULT_CLIENT_CERT" type="string">
    Path to the client certificate for mutual TLS authentication.
  </ParamField>

  <ParamField body="CONCOURSE_VAULT_CLIENT_KEY" type="string">
    Path to the client private key for mutual TLS authentication.
  </ParamField>

  <ParamField body="CONCOURSE_VAULT_SERVER_NAME" type="string">
    Override the SNI hostname sent during TLS negotiation. Useful when connecting via a load balancer.
  </ParamField>

  <ParamField body="CONCOURSE_VAULT_INSECURE_SKIP_VERIFY" type="boolean" default="false">
    Disable TLS certificate verification. **Never use in production.**
  </ParamField>
</Accordion>

<Accordion title="Auth settings">
  <ParamField body="CONCOURSE_VAULT_AUTH_BACKEND" type="string">
    Vault auth method to use. Common values: `token`, `approle`, `cert`, `ldap`, `github`. If unset, token auth is assumed.
  </ParamField>

  <ParamField body="CONCOURSE_VAULT_CLIENT_TOKEN" type="string">
    Static Vault token. Used only when `AUTH_BACKEND` is `token` or unset.
  </ParamField>

  <ParamField body="CONCOURSE_VAULT_CLIENT_TOKEN_PATH" type="string">
    Path to a file containing the Vault client token. Concourse re-reads this file on each renewal, which supports externally-rotated tokens.
  </ParamField>

  <ParamField body="CONCOURSE_VAULT_AUTH_PARAM" type="string">
    Key-value pair passed to the auth backend login endpoint. Specify multiple times for multiple params. Format: `key:value`. For AppRole: `role_id:xxx` and `secret_id:yyy`.
  </ParamField>

  <ParamField body="CONCOURSE_VAULT_AUTH_BACKEND_MAX_TTL" type="duration">
    Force a full re-login after this duration, regardless of token TTL. Useful for rotating secret IDs on a schedule.
  </ParamField>

  <ParamField body="CONCOURSE_VAULT_RETRY_MAX" type="duration" default="5m">
    Maximum backoff interval between retry attempts when login or re-auth fails.
  </ParamField>

  <ParamField body="CONCOURSE_VAULT_RETRY_INITIAL" type="duration" default="1s">
    Initial wait time before the first retry. Subsequent retries use exponential backoff up to `RETRY_MAX`.
  </ParamField>
</Accordion>

## Secret path conventions

Concourse constructs the full Vault path by combining the path prefix with lookup templates. With the default prefix `/concourse` the search order for a secret named `db-password` in team `main` and pipeline `myapp` is:

| Priority | Path | Scope |
|---|---|---|
| 1 | `/concourse/main/myapp/db-password` | Pipeline-scoped |
| 2 | `/concourse/main/db-password` | Team-scoped |
| 3 | `/concourse/<shared-path>/db-password` | Shared (if `SHARED_PATH` is set) |

<Note>
  If the secret has a `value` key (e.g. `vault kv put ... value="s3cr3t"`), Concourse extracts that single string. If the secret has multiple keys Concourse exposes the entire map, allowing field access with dot notation: `((db-creds.username))`.
</Note>

## Writing secrets to Vault

<Tabs>
  <Tab title="Simple string secret">
    ```bash
    vault kv put /concourse/main/myapp/database-password \
      value="s3cr3t"
    ```
  </Tab>
  <Tab title="Multi-field secret">
    ```bash
    vault kv put /concourse/main/myapp/database \
      username="concourse" \
      password="s3cr3t" \
      host="db.internal"
    ```

    Access individual fields in the pipeline with `((database.username))`, `((database.password))`, etc.
  </Tab>
  <Tab title="Team-scoped secret (shared across pipelines)">
    ```bash
    vault kv put /concourse/main/slack-webhook \
      value="https://hooks.slack.com/services/xxx"
    ```
  </Tab>
</Tabs>

## Using Vault secrets in a pipeline

Reference secrets anywhere a string value is accepted:

```yaml
resources:
  - name: production-db
    type: registry-image
    source:
      repository: myregistry.example.com/app
      username: ((registry-username))
      password: ((registry-password))

jobs:
  - name: deploy
    plan:
      - task: run-migration
        config:
          platform: linux
          image_resource:
            type: registry-image
            source: { repository: alpine }
          run:
            path: sh
            args: ["-c", "echo migrating"]
          params:
            DATABASE_URL: ((database.host))
            DATABASE_PASS: ((database.password))
```

## Using Vault as a pipeline var_source

Define a named Vault source inside the pipeline when you need to draw from a different Vault cluster or auth configuration than the globally configured one:

```yaml
var_sources:
  - name: prod-vault
    type: vault
    config:
      url: https://vault.example.com
      auth_backend: approle
      auth_params:
        role_id: ((vault-role-id))
        secret_id: ((vault-secret-id))
      path_prefix: /myorg

resources:
  - name: s3-release
    type: s3
    source:
      bucket: releases
      access_key_id: ((prod-vault:aws-access-key))
      secret_access_key: ((prod-vault:aws-secret-key))
```

<Tip>
  The `role_id` and `secret_id` values in the `var_source` config are themselves resolved from the **global** Vault backend before the per-pipeline source is initialised. This means you only need to store two bootstrap secrets globally to unlock an entire per-pipeline Vault configuration.
</Tip>

## Vault policy example

Grant Concourse's AppRole read access to the `/concourse` path:

```hcl
# concourse-policy.hcl
path "secret/data/concourse/*" {
  capabilities = ["read", "list"]
}

path "secret/metadata/concourse/*" {
  capabilities = ["read", "list"]
}
```

```bash
vault policy write concourse concourse-policy.hcl

vault auth enable approle
vault write auth/approle/role/concourse \
  token_policies="concourse" \
  token_ttl=1h \
  token_max_ttl=4h

vault read auth/approle/role/concourse/role-id
vault write -f auth/approle/role/concourse/secret-id
```
