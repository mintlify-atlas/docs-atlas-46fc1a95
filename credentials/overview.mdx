Concourse's credential management system lets you keep sensitive values — passwords, tokens, API keys — out of your pipeline definitions entirely. Instead of embedding secrets in YAML, you reference them with the `((var_name))` double-parentheses syntax. Concourse resolves each reference at runtime against a configured credentials backend, so the plain-text value is never written to the database, never visible in the Concourse UI, and redacted automatically from build logs.

## Variable interpolation syntax

Concourse supports three forms of variable reference in pipeline configs:

| Syntax | Meaning |
|---|---|
| `((secret-name))` | Look up `secret-name` in the globally configured backend using the default path hierarchy |
| `((source-name:secret-name))` | Look up `secret-name` in a named `var_source` defined in the same pipeline |
| `((.:local-var))` | Reference a local variable set by a preceding `load_var` step in the same build |

Variables can appear in **any string value** inside a pipeline definition — resource `source` blocks, task `params`, job `on_success` configs, and so on. Concourse evaluates them lazily: a pipeline referencing `((db-password))` is valid even if the secret does not yet exist in the backend, and will only fail at the point where a build actually tries to use it.

## Secret lookup path order

When Concourse resolves a plain `((secret-name))` reference it searches the backend in order, stopping at the first match:

1. **Pipeline-scoped path** — `/concourse/<team>/<pipeline>/<secret>`
2. **Team-scoped path** — `/concourse/<team>/<secret>`

This hierarchy is implemented in [`secret_var_lookup_rules.go`](https://github.com/concourse/concourse/blob/master/atc/creds/secret_var_lookup_rules.go) using Go text templates, which means both the pipeline and team path templates are fully customisable per backend. Some backends (Vault, AWS SSM) also support an optional **shared path** for secrets that should be accessible across all teams.

<Note>
  Variables are resolved **at runtime**, not when `fly set-pipeline` is called. The Concourse database stores only the raw `((var_name))` string, never the resolved secret value.
</Note>

## Default path templates

The default templates used by most backends follow a common convention:

```
/concourse/{{.Team}}/{{.Pipeline}}/{{.Secret}}   ← pipeline-scoped (checked first)
/concourse/{{.Team}}/{{.Secret}}                 ← team-scoped    (checked second)
```

The template variables `.Team`, `.Pipeline`, and `.Secret` are substituted by Concourse at lookup time. If a backend is queried outside of a pipeline context (for example in a `set_pipeline` step referencing a team-level credential) the pipeline-scoped template is skipped automatically.

## Named var_sources

You can define per-pipeline credential sources using the `var_sources` key at the top of a pipeline YAML. This lets different pipelines draw secrets from different Vault clusters, different AWS accounts, or any combination of backends — without reconfiguring the `web` node.

```yaml
var_sources:
  - name: prod-vault
    type: vault
    config:
      url: https://vault.example.com
      auth_backend: approle
      auth_params:
        role_id: ((vault-role-id))
        secret_id: ((vault-secret-id))
```

Reference secrets from a named source with the `source-name:secret-name` colon syntax:

```yaml
resources:
  - name: production-db
    type: git
    source:
      password: ((prod-vault:database-password))
```

<Tip>
  The `auth_params` values inside a `var_source` config are themselves interpolated from the globally configured backend, so you can bootstrap per-pipeline sources from a small number of global secrets.
</Tip>

## Secret redaction

Concourse automatically redacts resolved secret values from build output. Whenever a string returned by a credentials backend appears in a build's stdout or stderr stream, Concourse replaces it with `((redacted))` before the log is stored or streamed.

<Warning>
  Setting the environment variable `CONCOURSE_DISABLE_REDACT_SECRETS=true` on the `web` node disables redaction globally. Use this only for debugging in a non-production environment — secret values will appear in plain text in all build logs.
</Warning>

## Caching and retries

The `web` node wraps every backend call with two optional layers:

- **Retry** — controlled by `CONCOURSE_SECRET_RETRY_ATTEMPTS` and `CONCOURSE_SECRET_RETRY_INTERVAL`. Failed backend calls are retried before the build task fails.
- **Cache** — controlled by `CONCOURSE_SECRET_CACHE_ENABLED` and `CONCOURSE_SECRET_CACHE_DURATION`. Successfully resolved secrets are cached in memory on the `web` node to reduce backend traffic during high-parallelism builds.
