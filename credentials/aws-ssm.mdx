Concourse ships with native support for two AWS-managed secrets stores: **AWS Systems Manager (SSM) Parameter Store** and **AWS Secrets Manager**. Both backends follow the same path-template conventions as other Concourse credential managers, and both support IAM role-based authentication — meaning you don't need to configure static access keys when running on AWS infrastructure. The two backends are separate plugins; you configure one or the other (or both via `var_sources`) depending on which service you use.

---

## AWS SSM Parameter Store

AWS SSM Parameter Store stores secrets as named parameters. Concourse reads parameters using the `GetParameter` and `GetParametersByPath` API calls, automatically requesting decryption for `SecureString` values.

### Configuration

All settings are passed as environment variables to the `concourse web` process, or as equivalent `--aws-ssm-*` flags.

<ParamField body="CONCOURSE_AWS_SSM_REGION" type="string" required>
  AWS region to query. Example: `us-east-1`. This is the only required field when using IAM role authentication.
</ParamField>

<ParamField body="CONCOURSE_AWS_SSM_ACCESS_KEY" type="string">
  AWS access key ID. Leave unset to use the default AWS credential chain (instance profile, ECS task role, environment variables, `~/.aws/credentials`).
</ParamField>

<ParamField body="CONCOURSE_AWS_SSM_SECRET_KEY" type="string">
  AWS secret access key. Must be provided if `ACCESS_KEY` is set.
</ParamField>

<ParamField body="CONCOURSE_AWS_SSM_SESSION_TOKEN" type="string">
  AWS session token for temporary credentials obtained from STS. Used together with `ACCESS_KEY` and `SECRET_KEY`.
</ParamField>

<ParamField body="CONCOURSE_AWS_SSM_PIPELINE_SECRET_TEMPLATE" type="string" default="/concourse/{{.Team}}/{{.Pipeline}}/{{.Secret}}">
  Go template for the pipeline-scoped parameter path. The variables `.Team`, `.Pipeline`, and `.Secret` are substituted at lookup time.
</ParamField>

<ParamField body="CONCOURSE_AWS_SSM_TEAM_SECRET_TEMPLATE" type="string" default="/concourse/{{.Team}}/{{.Secret}}">
  Go template for the team-scoped parameter path. This path is tried when the pipeline-scoped path returns no result.
</ParamField>

<ParamField body="CONCOURSE_AWS_SSM_SHARED_PATH" type="string">
  Optional path prefix for parameters shared across all teams and pipelines. Checked last in the lookup order.
</ParamField>

### Secret lookup order

For a secret named `db-password` in team `main` and pipeline `myapp`, Concourse queries SSM in this order:

| Priority | Parameter name | Scope |
|---|---|---|
| 1 | `/concourse/main/myapp/db-password` | Pipeline-scoped |
| 2 | `/concourse/main/db-password` | Team-scoped |
| 3 | `<shared-path>/db-password` | Shared (if `SHARED_PATH` is set) |

<Note>
  Concourse first attempts `GetParameter` (exact name match). If that returns no result, it falls back to `GetParametersByPath` to support complex multi-key secrets stored as a parameter path hierarchy. The `value` of the first matching path entry is returned as a string; multiple sub-keys are returned as a map.
</Note>

### Writing parameters to SSM

<Tabs>
  <Tab title="Pipeline-scoped secret">
    ```bash
    aws ssm put-parameter \
      --name "/concourse/main/my-pipeline/db-password" \
      --value "s3cr3t" \
      --type SecureString \
      --region us-east-1
    ```
  </Tab>
  <Tab title="Team-scoped secret">
    ```bash
    aws ssm put-parameter \
      --name "/concourse/main/slack-webhook" \
      --value "https://hooks.slack.com/services/xxx" \
      --type SecureString \
      --region us-east-1
    ```
  </Tab>
  <Tab title="Multi-field secret (path hierarchy)">
    ```bash
    # Access as ((db.username)), ((db.password)), etc.
    aws ssm put-parameter \
      --name "/concourse/main/my-pipeline/db/username" \
      --value "concourse" \
      --type String \
      --region us-east-1

    aws ssm put-parameter \
      --name "/concourse/main/my-pipeline/db/password" \
      --value "s3cr3t" \
      --type SecureString \
      --region us-east-1
    ```
  </Tab>
</Tabs>

### IAM policy for Concourse

Grant Concourse's IAM role or user the minimum permissions needed to read parameters:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "ssm:GetParameter",
        "ssm:GetParametersByPath"
      ],
      "Resource": "arn:aws:ssm:us-east-1:123456789012:parameter/concourse/*"
    },
    {
      "Effect": "Allow",
      "Action": "kms:Decrypt",
      "Resource": "arn:aws:kms:us-east-1:123456789012:key/<your-kms-key-id>"
    }
  ]
}
```

<Tip>
  When running the Concourse `web` node on an EC2 instance, ECS task, or EKS pod with an appropriate IAM role attached, you can omit `CONCOURSE_AWS_SSM_ACCESS_KEY` and `CONCOURSE_AWS_SSM_SECRET_KEY` entirely. The AWS SDK picks up the role credentials automatically from the instance metadata service.
</Tip>

## AWS Secrets Manager

AWS Secrets Manager is an alternative to SSM Parameter Store, with built-in secret rotation, versioning, and cross-account sharing. Concourse's Secrets Manager backend uses the `GetSecretValue` API and supports the same path-template lookup model as SSM.

### Configuration

<ParamField body="CONCOURSE_AWS_SECRETSMANAGER_REGION" type="string" required>
  AWS region to query. Example: `eu-west-1`.
</ParamField>

<ParamField body="CONCOURSE_AWS_SECRETSMANAGER_ACCESS_KEY" type="string">
  AWS access key ID. Omit when using an IAM role.
</ParamField>

<ParamField body="CONCOURSE_AWS_SECRETSMANAGER_SECRET_KEY" type="string">
  AWS secret access key. Must be set if `ACCESS_KEY` is provided.
</ParamField>

<ParamField body="CONCOURSE_AWS_SECRETSMANAGER_SESSION_TOKEN" type="string">
  AWS session token for temporary STS credentials.
</ParamField>

<ParamField body="CONCOURSE_AWS_SECRETSMANAGER_PIPELINE_SECRET_TEMPLATE" type="string" default="/concourse/{{.Team}}/{{.Pipeline}}/{{.Secret}}">
  Go template for the pipeline-scoped secret identifier.
</ParamField>

<ParamField body="CONCOURSE_AWS_SECRETSMANAGER_TEAM_SECRET_TEMPLATE" type="string" default="/concourse/{{.Team}}/{{.Secret}}">
  Go template for the team-scoped secret identifier.
</ParamField>

<ParamField body="CONCOURSE_AWS_SECRETSMANAGER_SHARED_SECRET_TEMPLATE" type="string" default="/concourse/{{.Secret}}">
  Go template for secrets shared across all teams. Unlike SSM, Secrets Manager supports a third shared template out of the box.
</ParamField>

### Writing secrets to Secrets Manager

<Tabs>
  <Tab title="Plain string">
    ```bash
    aws secretsmanager create-secret \
      --name "/concourse/main/my-pipeline/github-token" \
      --secret-string "ghp_xxxxxxxxxxxx" \
      --region eu-west-1
    ```
  </Tab>
  <Tab title="JSON object (multi-field)">
    ```bash
    # Access individual fields as ((db.host)), ((db.password)), etc.
    aws secretsmanager create-secret \
      --name "/concourse/main/my-pipeline/db" \
      --secret-string '{"host":"db.internal","username":"app","password":"s3cr3t"}' \
      --region eu-west-1
    ```
  </Tab>
  <Tab title="Update an existing secret">
    ```bash
    aws secretsmanager put-secret-value \
      --secret-id "/concourse/main/my-pipeline/github-token" \
      --secret-string "ghp_newtoken" \
      --region eu-west-1
    ```
  </Tab>
</Tabs>

### IAM policy for Secrets Manager

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": "secretsmanager:GetSecretValue",
      "Resource": "arn:aws:secretsmanager:eu-west-1:123456789012:secret:/concourse/*"
    }
  ]
}
```

### Using IAM roles on AWS infrastructure

No key configuration is required when the `web` node runs inside AWS with an appropriate role:

<CodeGroup>
  ```yaml ECS Task Definition (excerpt)
  taskRoleArn: arn:aws:iam::123456789012:role/concourse-web-role
  containerDefinitions:
    - name: concourse-web
      environment:
        - name: CONCOURSE_AWS_SECRETSMANAGER_REGION
          value: eu-west-1
        # No ACCESS_KEY or SECRET_KEY — role credentials used automatically
  ```

  ```yaml EKS Pod (ServiceAccount annotation)
  apiVersion: v1
  kind: ServiceAccount
  metadata:
    name: concourse-web
    namespace: concourse
    annotations:
      eks.amazonaws.com/role-arn: arn:aws:iam::123456789012:role/concourse-web-role
  ```
</CodeGroup>

<Warning>
  Do not commit AWS access keys to source control or include them in container images. Use IAM roles or, at minimum, store the keys in a secrets manager and inject them via environment variables at runtime.
</Warning>

## SSM vs Secrets Manager — choosing the right backend

| | SSM Parameter Store | Secrets Manager |
|---|---|---|
| **Secret rotation** | Manual | Built-in, Lambda-based |
| **Versioning** | Labels only | Full version history |
| **Pricing** | Free for `String`/`SecureString` standard tier | Per-secret monthly fee + API call charges |
| **Cross-account sharing** | Limited | Native via resource policies |
| **Max secret size** | 8 KB | 65 KB |
| **Concourse shared template** | Via `SHARED_PATH` | Via `SHARED_SECRET_TEMPLATE` |
