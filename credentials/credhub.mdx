Concourse ships with two additional built-in credential backends suited for different deployment environments: **CredHub**, the credential store from the CloudFoundry ecosystem, and **Kubernetes Secrets**, a natural fit for Concourse installations running inside a Kubernetes cluster. Both backends follow the same path hierarchy and runtime-resolution model as all other Concourse credential managers — secrets are never persisted to the Concourse database and are redacted from build logs automatically.

---

## CredHub

CredHub is the credential management service used across Cloud Foundry and BOSH deployments. It supports several credential types — passwords, certificates, RSA keys, SSH keys, and arbitrary JSON values — and uses UAA (User Account and Authentication) for access control. Concourse authenticates to CredHub with a UAA client credential and uses `FindByPartialName` followed by `GetLatestVersion` to resolve secrets at build time.

### Configuration

All settings are passed as environment variables to the `concourse web` process, or as equivalent `--credhub-*` flags.

<ParamField body="CONCOURSE_CREDHUB_URL" type="string" required>
  Full URL of the CredHub server. Example: `https://credhub.example.com:8844`
</ParamField>

<ParamField body="CONCOURSE_CREDHUB_PATH_PREFIX" type="string" default="/concourse">
  Path under which all credential lookups are namespaced. Concourse appends the team, pipeline, and secret name to this prefix.
</ParamField>

<ParamField body="CONCOURSE_CREDHUB_CLIENT_ID" type="string">
  UAA OAuth2 client ID used to authenticate Concourse to CredHub. The UAA client must have the `credhub.read` authority.
</ParamField>

<ParamField body="CONCOURSE_CREDHUB_CLIENT_SECRET" type="string">
  UAA OAuth2 client secret corresponding to `CLIENT_ID`.
</ParamField>

<ParamField body="CONCOURSE_CREDHUB_CA_CERT" type="string">
  Path to one or more PEM-encoded CA certificate files (comma-separated or flag repeated) used to verify the CredHub server's TLS certificate. Required when CredHub uses a private CA, which is typical in BOSH-deployed environments.
</ParamField>

<ParamField body="CONCOURSE_CREDHUB_CLIENT_CERT" type="string">
  Path to a client certificate for mutual TLS authentication against CredHub.
</ParamField>

<ParamField body="CONCOURSE_CREDHUB_CLIENT_KEY" type="string">
  Path to the private key corresponding to `CLIENT_CERT`.
</ParamField>

<ParamField body="CONCOURSE_CREDHUB_INSECURE_SKIP_VERIFY" type="boolean" default="false">
  Skip TLS certificate verification. **Not recommended in production** — use `CA_CERT` instead.
</ParamField>

### Secret path conventions

CredHub paths are constructed from the prefix, team name, pipeline name, and secret name. With the default prefix `/concourse`, the lookup order for a secret `github-token` in team `main` and pipeline `my-pipeline` is:

| Priority | CredHub path | Scope |
|---|---|---|
| 1 | `/concourse/main/my-pipeline/github-token` | Pipeline-scoped |
| 2 | `/concourse/main/github-token` | Team-scoped |

<Note>
  CredHub's `FindByPartialName` is used to check existence before `GetLatestVersion` fetches the value. This means Concourse always retrieves the **latest version** of a credential — CredHub's version history is transparent to the pipeline author.
</Note>

### Writing credentials to CredHub

<Tabs>
  <Tab title="Simple value">
    ```bash
    credhub set \
      --name /concourse/main/my-pipeline/github-token \
      --type value \
      --value ghp_xxxxxxxxxxxx
    ```
  </Tab>
  <Tab title="Username/password">
    ```bash
    credhub set \
      --name /concourse/main/my-pipeline/db-creds \
      --type user \
      --username concourse \
      --password s3cr3t
    ```

    Access with `((db-creds.username))` and `((db-creds.password))`.
  </Tab>
  <Tab title="Certificate">
    ```bash
    credhub generate \
      --name /concourse/main/my-pipeline/tls-cert \
      --type certificate \
      --common-name myapp.example.com \
      --ca /your-ca
    ```

    Access with `((tls-cert.certificate))`, `((tls-cert.private_key))`, and `((tls-cert.ca))`.
  </Tab>
  <Tab title="Team-scoped credential">
    ```bash
    credhub set \
      --name /concourse/main/slack-webhook \
      --type value \
      --value "https://hooks.slack.com/services/xxx"
    ```
  </Tab>
</Tabs>

### Using CredHub in a pipeline

```yaml
resources:
  - name: app-repo
    type: git
    source:
      uri: https://github.com/myorg/app.git
      private_key: ((git-deploy-key.private_key))

jobs:
  - name: build
    plan:
      - get: app-repo
      - task: test
        config:
          platform: linux
          image_resource:
            type: registry-image
            source: { repository: golang, tag: "1.22" }
          params:
            GITHUB_TOKEN: ((github-token))
          run:
            path: go
            args: [test, ./...]
```

### UAA client setup (BOSH / CF)

```bash
# Create a UAA client for Concourse with credhub.read scope
uaac client add concourse-client \
  --name concourse-client \
  --secret "your-client-secret" \
  --authorized_grant_types client_credentials \
  --authorities credhub.read
```

<Warning>
  The UAA client secret is itself a credential — store it securely and rotate it on a schedule. Do not hard-code it in BOSH manifests or Concourse deployment configs without an external secrets mechanism.
</Warning>

## Kubernetes Secrets

The Kubernetes Secrets backend is the natural choice when Concourse is deployed inside a Kubernetes cluster. It maps each Concourse team to a Kubernetes namespace (prefixed with `concourse-` by default) and reads Kubernetes `Secret` objects by name. No external service is required — the backend calls the Kubernetes API directly, either using the `web` pod's in-cluster service account or an external kubeconfig.

### Configuration

<ParamField body="CONCOURSE_KUBERNETES_IN_CLUSTER" type="boolean" default="false">
  Use the in-cluster service account token and CA certificate mounted automatically by Kubernetes. Enable this when the `web` pod runs inside the same cluster that holds the secrets.
</ParamField>

<ParamField body="CONCOURSE_KUBERNETES_CONFIG_PATH" type="string">
  Path to a kubeconfig file for out-of-cluster access. Cannot be used together with `IN_CLUSTER`.
</ParamField>

<ParamField body="CONCOURSE_KUBERNETES_NAMESPACE_PREFIX" type="string" default="concourse-">
  Prefix prepended to team names to form the Kubernetes namespace. With the default `concourse-`, the team `main` maps to namespace `concourse-main`.
</ParamField>

<Note>
  `IN_CLUSTER` and `CONFIG_PATH` are mutually exclusive. Providing both causes Concourse to return a validation error on startup.
</Note>

### Secret naming convention

Kubernetes Secrets are looked up by the following rules:

| Concourse concept | Kubernetes object |
|---|---|
| Team `main` | Namespace `concourse-main` |
| Secret `github-token` | `Secret` named `github-token` in `concourse-main` |
| Pipeline-scoped secret `myapp.github-token` | `Secret` named `myapp.github-token` in `concourse-main` |

Concourse reads the `value` key from the `Secret`'s `data` map for simple string secrets. If no `value` key exists, it returns all keys in the `data` map as a structured object, allowing field access with dot notation: `((db-creds.username))`.

### Creating Kubernetes Secrets for Concourse

<Tabs>
  <Tab title="Simple string (kubectl)">
    ```bash
    kubectl create namespace concourse-main

    kubectl create secret generic github-token \
      --namespace concourse-main \
      --from-literal=value=ghp_xxxxxxxxxxxx
    ```
  </Tab>
  <Tab title="Multi-field secret">
    ```bash
    kubectl create secret generic db-creds \
      --namespace concourse-main \
      --from-literal=username=concourse \
      --from-literal=password=s3cr3t \
      --from-literal=host=db.internal
    ```

    Reference individual fields with `((db-creds.username))`, `((db-creds.password))`, etc.
  </Tab>
  <Tab title="Pipeline-scoped secret">
    ```bash
    # Accessible only to the 'myapp' pipeline in team 'main'
    kubectl create secret generic myapp.deploy-key \
      --namespace concourse-main \
      --from-file=value=/path/to/private_key
    ```
  </Tab>
  <Tab title="YAML manifest">
    ```yaml
    apiVersion: v1
    kind: Secret
    metadata:
      name: github-token
      namespace: concourse-main
    type: Opaque
    stringData:
      value: ghp_xxxxxxxxxxxx
    ```
  </Tab>
</Tabs>

### RBAC for the Concourse web pod

Grant the `web` pod's service account read access to secrets in the `concourse-*` namespaces:

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: concourse-secret-reader
rules:
  - apiGroups: [""]
    resources: ["secrets"]
    verbs: ["get", "list"]
---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: concourse-secret-reader
  namespace: concourse-main        # Repeat per team namespace
roleRef:
  apiGroup: rbac.authorization.k8s.io
  kind: ClusterRole
  name: concourse-secret-reader
subjects:
  - kind: ServiceAccount
    name: concourse-web
    namespace: concourse
```

<Tip>
  Scope the `RoleBinding` to each team's namespace rather than using a `ClusterRoleBinding`. This enforces namespace isolation so the `web` pod can only read secrets in namespaces explicitly granted, matching Concourse's team-level access model.
</Tip>

## CyberArk Conjur

Concourse also includes a built-in backend for **CyberArk Conjur**, an enterprise secrets management platform popular in regulated industries. Conjur is configured via `CONCOURSE_CONJUR_*` environment variables, supports host-based API key authentication and token file authentication (for Kubernetes and IAM workloads), and uses the same `concourse/{{.Team}}/{{.Pipeline}}/{{.Secret}}` path template convention. See [`atc/creds/conjur`](https://github.com/concourse/concourse/tree/master/atc/creds/conjur) in the source tree for the full list of configuration options.
